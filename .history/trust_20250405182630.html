1. Password Handling is 100% Local

You can verify that Skeb Sniper never transmits your Twitter credentials and encrypts them locally:

  1. Block all internet access for Skeb Sniper using your firewall or network monitor.
  2. Launch Skeb Sniper and enter a random Twitter username and password.
  3. Locate the encrypted credentials file created inside Skeb Sniper’s installation directory.
  4. Open the encrypted file with a text editor — you will see unreadable encrypted data.
  5. Monitor your network activity — you will see no outgoing connections during password entry.

Result: This proves that password handling is performed purely locally, without transmitting any sensitive information.

2. User and Machine-Specific Encryption

You can verify that even if the encrypted credentials file is stolen, it cannot be decrypted on any other machine or user account:

  1. Copy the encrypted credentials file to another user account on the same machine.
  2. Launch Skeb Sniper under the new user account and attempt to decrypt the file.
  3. Observe that decryption fails because the encryption key is different.
  4. Copy the encrypted file to a different machine entirely and try again.
  5. Observe that decryption also fails on a different machine.

Result: This proves that Skeb Sniper's encryption is tied to both your specific device and user account, preventing unauthorized decryption even if the file is stolen.

  3. Send a test email to your Gmail account.
  4. Observe that Skeb Sniper still detects the new email correctly, even with our server blocked.
